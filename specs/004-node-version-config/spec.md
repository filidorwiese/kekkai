# Feature Specification: Node Version Config

**Feature Branch**: `004-node-version-config`

**Created**: 2026-07-07

**Status**: Draft

**Input**: User description: "since a user cant actually change the base_image (it's needs to be a derivative of debian and node) i want to change the configuration image.base_image to image.node_version (which defaults to lts). A user could choose another version of node - but it needs to exists as a base image. We suffix the image with `-trixie` internally to ensure a pinned debian version"

## Clarifications

### Session 2026-07-07

- Q: Should the system verify the configured Node version exists as a base image before pulling? → A: Best-effort pre-check — skip if image already available locally; otherwise query the registry before any build/pull; missing tag → clean error naming the version; registry unreachable → skip the check and let the pull surface the failure.
- Q: Which artifacts must drop the legacy `image.base_image` references? → A: README.md, SPECIFICATION.md, and the init-generated starter config — all references removed, replaced by `image.node_version`.

## User Scenarios & Testing *(mandatory)*

### User Story 1 - Configure only the Node version (Priority: P1)

A user wants the sandbox to run a specific Node.js version. Instead of writing a full
container image reference (which invites invalid choices, since the image must be a
Debian-based Node image anyway), they set a single `image.node_version` value such as
`22` or `lts`. The system resolves that to the correct pinned Debian-based image
internally. When the key is omitted, the sandbox uses the current LTS version of Node.

**Why this priority**: This is the feature itself — replacing a misleadingly flexible
setting with one that only exposes the choice users can actually make.

**Independent Test**: Set `image.node_version: 22` in a project config, start the
sandbox, and verify it runs Node 22 on the pinned Debian release. Remove the key and
verify the sandbox runs the LTS version.

**Acceptance Scenarios**:

1. **Given** a config with no `image` section, **When** the sandbox starts, **Then** it runs the LTS version of Node on the pinned Debian release.
2. **Given** a config with `image.node_version: 22`, **When** the sandbox starts, **Then** it runs Node 22 on the pinned Debian release.
3. **Given** a config with `image.node_version: 22`, **When** the user later changes it to `24`, **Then** the next sandbox start uses Node 24 (the version change is picked up as an image input change).

---

### User Story 2 - Clear migration from the old key (Priority: P2)

A user with an existing config that still sets `image.base_image` runs the tool after
upgrading. Instead of a bare "unknown key" failure, they get a targeted message telling
them the key was replaced by `image.node_version` and how to migrate.

**Why this priority**: Existing configs (including those generated by the tool's own
init command) use the old key; without a guided error, every existing user hits a
confusing failure.

**Independent Test**: Run the tool in a project whose config sets
`image.base_image: node:24-trixie` and verify the error names both the old and the new
key.

**Acceptance Scenarios**:

1. **Given** a config containing `image.base_image`, **When** validation runs, **Then** the user sees a migration error pointing to `image.node_version`, reported in the same single validation pass as any other config violations.
2. **Given** a freshly initialized project, **When** the user inspects the generated config, **Then** it documents `image.node_version` (default `lts`) and does not mention `image.base_image`.

---

### User Story 3 - Helpful failure for a non-existent version (Priority: P3)

A user sets `image.node_version` to a value for which no published Node base image
exists (for example a typo like `2x` or a version never published for the pinned Debian
release). The system rejects malformed values before doing any container work, and for
well-formed versions performs a best-effort existence check against the image registry
before any build or pull starts, failing with an error that clearly attributes the
problem to the configured Node version.

**Why this priority**: The value is free-form user input; without guardrails a typo
produces an opaque container-tooling error late in the startup flow.

**Independent Test**: Set `image.node_version: "not a version"` and verify config
validation rejects it; set a well-formed but unpublished version and verify startup
fails before any build/pull with an error naming the configured version.

**Acceptance Scenarios**:

1. **Given** `image.node_version` containing characters that cannot form a valid version selector (spaces, `:`, `/`, uppercase), **When** validation runs, **Then** the config is rejected with a message naming the offending value.
2. **Given** a well-formed version with no published base image and a reachable registry, **When** startup reaches image acquisition, **Then** the pre-check fails before any build/pull with an error telling the user the configured Node version has no available base image.
3. **Given** the required base image is already available locally, **When** startup runs, **Then** no registry existence check is performed.
4. **Given** the registry is unreachable and the image is not local, **When** startup runs, **Then** the pre-check is skipped and any failure surfaces from the normal pull attempt.

---

### Edge Cases

- User supplies a full image reference (e.g. `node:24-trixie`) in `image.node_version`: rejected by format validation (contains `:`), with the message making clear only a version selector is expected.
- User supplies a value already carrying the Debian suffix (e.g. `24-trixie`): treated as an invalid version selector so the internal suffixing never doubles up.
- Named channels like `lts` or `current` move over time upstream: the configured string is the image input; the system does not re-resolve or auto-refresh when the channel advances (same behavior as any mutable tag today).
- Empty string for `image.node_version`: rejected by validation rather than silently falling back to the default, since an explicit empty value signals a config mistake.
- Registry unreachable (offline, proxy, private mirror) while the image is already local: existence pre-check must not run or fail; the sandbox starts from the local image.

## Requirements *(mandatory)*

### Functional Requirements

- **FR-001**: The configuration MUST accept an optional `image.node_version` value selecting the Node.js version for the sandbox.
- **FR-002**: When `image.node_version` is omitted, the system MUST default to the `lts` channel.
- **FR-003**: The system MUST resolve the configured version to a Debian-based Node image pinned to the project's chosen Debian release (currently "trixie") by suffixing internally; users never specify the Debian portion.
- **FR-004**: The configuration MUST no longer accept `image.base_image`; the key MUST be treated as a legacy key producing a targeted migration error naming `image.node_version`, consistent with existing legacy-key handling.
- **FR-005**: Validation MUST reject values that cannot be a version selector (empty, whitespace, or containing image-reference syntax such as `:` or `/`) before any container work starts, reported in the same single-pass validation as other config errors.
- **FR-006**: Before building or pulling, the system MUST perform a best-effort existence check of the resolved base image against the registry: skipped when the image is already available locally; a confirmed-missing tag MUST fail with an error naming the configured Node version; an unreachable registry MUST NOT block startup (the check is skipped and pull-time errors apply).
- **FR-006a**: Any image-acquisition failure caused by a non-existent version (pre-check or pull) MUST name the configured Node version as the cause.
- **FR-007**: README.md, SPECIFICATION.md, and the init-generated starter config MUST contain no remaining reference to `image.base_image` (outside the legacy-key migration documentation, if any) and MUST describe `image.node_version` with its `lts` default.
- **FR-008**: Changing `image.node_version` MUST count as an image input change (a different version yields a different sandbox image), while leaving it unchanged MUST NOT trigger a rebuild.

### Key Entities

- **Node version selector**: The user-supplied value in `image.node_version` — a major version (`24`), a full version (`24.3.0`), or a named channel (`lts`, `current`). It is the only image-related choice exposed to users.
- **Resolved base image**: The internal image reference derived from the selector plus the pinned Debian suffix; never user-visible in config, only in diagnostics.

## Success Criteria *(mandatory)*

### Measurable Outcomes

- **SC-001**: A user can pick a Node version by editing exactly one config value, with no knowledge of container image naming required.
- **SC-002**: A project with no `image` configuration at all starts successfully on the LTS Node version.
- **SC-003**: 100% of configs still using the old key fail with a message that names the replacement key — zero bare "unknown key" failures for this migration.
- **SC-004**: Every malformed version value is caught during config validation, before any container work begins.
- **SC-005**: With a reachable registry, a well-formed but non-existent version fails before any image download starts, with an error naming the configured version; an unreachable registry never blocks startup when the image is already available locally.

## Assumptions

- The pinned Debian release name ("trixie") is an internal project constant, not user-configurable; bumping it is a separate future change.
- `image.base_image` gets a hard migration error rather than silent aliasing, matching the project's established legacy-key convention.
- Config validation only guards format; the existence pre-check is a startup step that runs after validation, since it requires network/registry access.
- Named channels (`lts`, `current`) are passed through as-is; the system does not pin them to a concrete version, accepting the same mutable-tag semantics the old default already had.
- The tool's own project documentation and design specification (README.md, SPECIFICATION.md) are updated alongside the change, per the project's spec-first principle; docs examples (e.g. sample project configs) follow the same sweep.
